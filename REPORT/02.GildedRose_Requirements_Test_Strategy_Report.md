# Gilded Rose 요구사항 및 테스트 전략 보고서

작성일: 2026-05-19  
작성 관점: 시니어 Java QA 엔지니어  
대상 프로젝트: Gilded Rose Java 프로젝트 (Java 21, Maven, JUnit 5)

## 1. 보고서 목적

본 보고서는 Gilded Rose 레거시 코드의 핵심 비즈니스 규칙을 QA 관점에서 정리하고, 리팩토링 및 신규 요구사항인 Conjured 아이템 구현 전에 확보해야 할 테스트 범위를 정의하기 위해 작성되었습니다.

현재 핵심 로직은 `GildedRose.updateQuality()`에 집중되어 있으며, 아이템 타입에 따라 `sellIn`과 `quality` 변경 규칙이 달라집니다. 따라서 테스트는 아이템 타입별 규칙, 경계값, 예외 조건을 중심으로 설계해야 합니다.

## 2. 핵심 비즈니스 규칙 요약

| 아이템 타입 | `sellIn` 규칙 | `quality` 규칙 | 만료 후 동작 | QA 우선순위 |
|---|---|---|---|---|
| Normal | 매일 1 감소 | 매일 1 감소 | 하루 총 2 감소 | 높음 |
| Aged Brie | 매일 1 감소 | 매일 1 증가 | 하루 총 2 증가 | 높음 |
| Backstage Pass | 매일 1 감소 | 날짜 구간별 1/2/3 증가 | 공연 이후 0 | 높음 |
| Sulfuras | 변경 없음 | 변경 없음 | 변경 없음 | 높음 |
| Conjured | 매일 1 감소 | 일반 아이템의 2배 감소 | 만료 후 하루 총 4 감소 | 매우 높음 |

## 3. 주요 리스크

1. `quality` 경계값 누락으로 0 미만 또는 50 초과 값이 발생할 수 있습니다.
2. `sellIn = 0`에서 업데이트 후 `sellIn = -1`이 되므로, 만료 후 규칙 적용 시점이 혼동될 수 있습니다.
3. `Backstage Pass`는 10일 이하, 5일 이하, 공연 이후 조건이 겹치므로 경계값 테스트가 부족하면 회귀가 발생하기 쉽습니다.
4. `Sulfuras`는 `quality` 상한 예외이면서 `sellIn`도 변하지 않아야 하므로 일반 공통 로직에 포함되면 결함이 발생할 수 있습니다.
5. `Conjured`는 신규 요구사항이며 현재 레거시 코드에서는 일반 아이템처럼 처리될 가능성이 있어, 구현 전 실패 테스트가 필요합니다.

## 4. 경계값 및 예외 조건

| 조건 | 기대 동작 | 테스트 필요성 |
|---|---|---|
| `quality = 0` | 감소형 아이템은 0 미만으로 내려가지 않음 | 하한 보장 |
| `quality = 50` | 증가형 아이템은 50 초과 불가 | 상한 보장 |
| `sellIn = 0` | 업데이트 후 -1이 되며 만료 후 규칙 적용 | 만료 전환점 검증 |
| `sellIn = -1` | 이미 만료된 규칙이 계속 적용 | 음수 sellIn 허용 검증 |
| Sulfuras `quality = 80` | 일반 상한 예외로 값 유지 | 예외 규칙 검증 |

## 5. Conjured 신규 요구사항

Conjured 아이템은 일반 아이템보다 `quality`가 2배 빠르게 감소해야 합니다.

1. 판매 기한 전에는 하루에 `quality`가 2 감소합니다.
2. 판매 기한이 지난 후에는 하루에 `quality`가 4 감소합니다.
3. `sellIn`은 일반 아이템과 동일하게 매일 1 감소합니다.
4. `quality`는 0 미만으로 내려가면 안 됩니다.
5. Sulfuras, Aged Brie, Backstage Pass와 명확히 구분되는 식별 규칙이 필요합니다.

## 6. 권장 테스트 전략

1. 먼저 현재 동작을 보호하는 Characterization Test를 작성합니다.
2. 아이템 타입별 규칙은 JUnit 5 `@ParameterizedTest`로 입력 조합을 명확히 검증합니다.
3. 모든 테스트는 `sellIn`과 `quality`를 함께 검증합니다.
4. 경계값은 최소한 `quality` 0/50, `sellIn` 0/-1을 포함합니다.
5. Conjured는 신규 요구사항이므로 구현 전 실패하는 테스트를 먼저 추가합니다.
6. 리팩토링은 전체 Maven 테스트가 Green인 상태에서만 진행합니다.

## 7. 우선 테스트 시나리오

1. Normal 아이템의 기본 감소 동작을 검증합니다.
2. Normal 아이템의 만료 후 2배 감소 동작을 검증합니다.
3. Normal 아이템의 `quality = 0` 하한을 검증합니다.
4. Aged Brie의 기본 증가 동작을 검증합니다.
5. Aged Brie의 만료 후 2배 증가 동작을 검증합니다.
6. Aged Brie의 `quality = 50` 상한을 검증합니다.
7. Backstage Pass의 10일 초과 구간 증가량을 검증합니다.
8. Backstage Pass의 10일 이하 구간 증가량을 검증합니다.
9. Backstage Pass의 5일 이하 구간 증가량을 검증합니다.
10. Backstage Pass의 공연 이후 `quality = 0` 동작을 검증합니다.
11. Sulfuras의 `sellIn` 불변을 검증합니다.
12. Sulfuras의 `quality` 불변을 검증합니다.
13. Sulfuras의 `quality = 80` 예외를 검증합니다.
14. Conjured의 판매 기한 전 2배 감소를 검증합니다.
15. Conjured의 만료 후 4 감소를 검증합니다.
16. Conjured의 `quality = 0` 하한을 검증합니다.
17. 여러 아이템이 함께 있을 때 각 규칙이 독립적으로 적용되는지 검증합니다.

## 8. 결론

Gilded Rose 프로젝트의 테스트 보강은 아이템 타입별 규칙과 경계값을 중심으로 진행해야 합니다. 특히 `sellIn = 0`, `sellIn = -1`, `quality = 0`, `quality = 50`은 회귀 가능성이 높은 지점이므로 필수 검증 대상입니다.

Conjured 요구사항은 기존 동작 변경을 수반하는 신규 기능이므로, 구현 전에 실패 테스트를 먼저 작성하고 Green 상태를 확인한 뒤 리팩토링을 진행하는 것이 가장 안전합니다.
