# Gilded Rose Golden Master 회귀 테스트 구현 보고서

작성 관점: 회귀 테스트(Approval/Golden Master) 설계 전문가  
대상 프로젝트: Gilded Rose Java 프로젝트  
기술 스택: Java 21, Maven, JUnit 5

## 1. 작업 목적

`TexttestFixture.java`의 콘솔 출력 결과를 기준 출력으로 고정하여, 향후 `GildedRose.updateQuality()` 리팩토링이나 규칙 변경 중 의도하지 않은 회귀를 빠르게 탐지할 수 있는 Golden Master 테스트를 구현했다.

이번 테스트는 기존 아이템별 단위 테스트를 대체하지 않고 보완한다. 단위 테스트가 개별 규칙과 경계값을 검증한다면, Golden Master 테스트는 여러 아이템을 30일 동안 연속 업데이트했을 때의 전체 출력 흐름을 한 번에 보호한다.

## 2. 구현 요약

주요 반영 사항:

- `TexttestFixture`의 출력 로직을 테스트에서 재사용할 수 있도록 메서드로 분리
- 현재 동작 기준의 30일 시뮬레이션 출력을 `golden_master_expected.txt`로 저장
- JUnit 5 테스트에서 actual 출력과 기준 파일을 문자열 비교
- Windows와 CI 환경의 줄바꿈 차이를 흡수하기 위해 line ending 정규화 적용
- Maven Surefire 설정을 추가하여 `mvn test`에서 자동 실행되도록 구성

## 3. 수정 및 추가 파일

### `src/test/java/com/gildedrose/TexttestFixture.java`

기존 `main()` 내부에 직접 있던 출력 로직을 다음 구조로 분리했다.

- `main(String[] args)`: 기존 CLI 실행 방식 유지
- `generateReport(int days)`: 테스트에서 출력 전체를 문자열로 획득
- `printReport(PrintStream out, int days)`: 실제 시뮬레이션 출력 생성

이 방식은 기존 `TexttestFixture 30` 실행 흐름을 유지하면서도, 테스트에서는 콘솔 캡처 없이 동일한 출력을 직접 비교할 수 있게 한다.

### `src/test/resources/golden_master_expected.txt`

현재 구현 기준의 Golden Master 파일이다.

보관 기준:

- 위치: `src/test/resources/golden_master_expected.txt`
- 내용: `TexttestFixture 30` 실행 결과
- 범위: day 0부터 day 30까지 총 31개 스냅샷
- 인코딩: UTF-8

기준 출력은 테스트 리소스로 보관하므로 Maven test phase에서 자동으로 `target/test-classes`에 복사된다.

### `src/test/java/com/gildedrose/GoldenMasterTest.java`

JUnit 5 기반 Golden Master 비교 테스트를 추가했다.

검증 흐름:

1. 클래스패스에서 `/golden_master_expected.txt`를 읽는다.
2. `TexttestFixture.generateReport(31)`로 actual 출력을 생성한다.
3. expected와 actual의 줄바꿈을 정규화한다.
4. `assertEquals`로 전체 문자열을 비교한다.

`TexttestFixture`의 CLI 인자 `30`은 내부적으로 `31`일 출력으로 변환되므로, 테스트에서는 `31`을 명시해 day 0부터 day 30까지 비교한다.

### `pom.xml`

Java 21 프로젝트 기준으로 Maven 컴파일 설정을 정리하고, JUnit 5 테스트가 Surefire에서 안정적으로 실행되도록 플러그인을 명시했다.

반영 내용:

- `maven.compiler.release`를 `21`로 설정
- `maven-compiler-plugin` 버전 명시
- `maven-surefire-plugin` 버전 명시
- `approvaltests` 의존성을 test scope로 제한

## 4. 기준 출력 생성 및 갱신 방법

Golden Master 파일은 의도된 동작 변경이 있을 때만 갱신한다. 단순 리팩토링 중 테스트가 실패한다면 기준 파일을 갱신하지 말고 구현 회귀 여부를 먼저 확인해야 한다.

PowerShell 기준 재생성 명령:

```powershell
mvn -q -DskipTests test-compile
$text = & java -cp "target/classes;target/test-classes" com.gildedrose.TexttestFixture 30 | Out-String
[System.IO.File]::WriteAllText("src/test/resources/golden_master_expected.txt", $text, [System.Text.UTF8Encoding]::new($false))
```

갱신 후에는 반드시 다음 명령으로 전체 테스트를 재실행한다.

```bash
mvn test
```

## 5. CI 자동 실행 구성

별도 CI 스크립트 변경 없이 Maven 표준 test phase에서 실행된다.

CI 실행 명령:

```bash
mvn test
```

자동 실행 근거:

- 테스트 클래스명이 `GoldenMasterTest`로 Surefire 기본 탐색 규칙에 포함됨
- 기준 파일이 `src/test/resources`에 있어 test resources 단계에서 클래스패스로 복사됨
- `maven-surefire-plugin`이 JUnit Platform provider를 사용해 JUnit 5 테스트를 실행함

## 7. Approvals 테스트 적용 가능성

현재 `pom.xml`에는 `approvaltests` 의존성이 포함되어 있으므로 ApprovalTests 방식도 적용 가능하다.

예시 방향:

```java
@Test
void thirtyDaySimulationApproval() {
    Approvals.verify(TexttestFixture.generateReport(31));
}
```

다만 이번 구현에서는 명시적인 파일 비교 방식을 선택했다.

선택 이유:

- CI에서 별도 reporter 설정 없이 안정적으로 동작
- 기준 파일 위치와 갱신 절차가 명확함
- 실패 시 JUnit 문자열 비교 결과로 expected/actual 차이를 바로 확인 가능
- 현재 프로젝트의 학습 및 보고서 목적에 더 직관적임

ApprovalTests는 향후 HTML diff, 전용 reporter, approved/received 파일 워크플로가 필요한 경우 도입하면 적합하다.

## 8. 운영 가이드

Golden Master 테스트 운영 원칙:

1. 리팩토링 중 Golden Master 실패는 회귀 가능성으로 간주한다.
2. 요구사항 변경 또는 결함 수정으로 출력 변화가 의도된 경우에만 기준 파일을 갱신한다.
3. 기준 파일 갱신 시 변경 diff를 리뷰하여 어떤 날짜와 아이템 출력이 바뀌었는지 확인한다.
4. Golden Master는 전체 흐름 보호용이므로, 세부 규칙 검증은 기존 `GildedRoseTest`의 아이템별 단위 테스트와 함께 유지한다.
