# Gilded Rose 결함 목록 보고서

작성 관점: QA 리드  
대상 프로젝트: Gilded Rose Java 프로젝트  
산출물: `docs/defect_list.md`

## 1. 작업 요약

현재까지 발견된 테스트 실패와 결함을 추적 가능한 목록 형태로 정리했다.

요청된 결함 항목 형식:

- `[ID]`
- `[Severity]`
- `[ItemType]`
- `[Steps]`
- `[Expected]`
- `[Actual]`
- `[Root Cause]`
- `[Fix Summary]`

위 형식에 맞춰 `docs/defect_list.md`를 생성했다.

## 2. 결함 현황

현재까지 재현된 테스트 실패 또는 확정 결함은 없다.

최신 테스트 실행 결과:

```text
Tests run: 28, Failures: 0, Errors: 0, Skipped: 0
BUILD SUCCESS
```

결함 목록에는 결함 없음 상태를 추적하기 위해 `DEF-000` Info 항목을 기록했다.

| ID | Severity | ItemType | 요약 |
| --- | --- | --- | --- |
| DEF-000 | Info | All | `mvn test` 기준 전체 테스트 Green, 재현 결함 없음 |

## 3. 작성 파일

### `docs/defect_list.md`

Markdown 표 형식으로 결함 목록을 작성했다.

포함 항목:

- 결함 목록 요약
- 최신 테스트 실행 결과
- 요청된 8개 필드 기반 결함 테이블
- 후속 실패 로그 수신 시 업데이트 기준

## 4. QA 판단

현재 상태는 결함 미재현으로 판단한다.

근거:

- 실패 로그가 제공되지 않았다.
- 로컬 `mvn test` 실행 결과 전체 테스트가 통과했다.
- `GildedRoseTest.java`의 28개 케이스가 Normal, Aged Brie, Backstage Pass, Sulfuras, Conjured 규칙을 모두 검증한다.
- 현재 `GildedRose.java` 구현은 테스트 기대값을 만족한다.

## 5. 후속 관리 기준

추가 실패 로그가 확인되면 `docs/defect_list.md`에 신규 결함 ID를 추가한다.

권장 ID 체계:

- `DEF-001`, `DEF-002`, ...

신규 결함 등록 시 필수 기록:

- 실패 재현 절차
- 기대값과 실제값
- 영향을 받는 ItemType
- `GildedRose.java` 기준 Root Cause
- 최소 수정 요약
- 수정 후 `mvn test` 결과
