# Gilded Rose QA/Refactoring Report

작성일: 2026-05-19

## 1. 프로젝트 개요

이 프로젝트는 Gilded Rose 레거시 코드 카타를 기반으로 한 Java/Maven 프로젝트입니다. 핵심 로직은 `GildedRose.updateQuality()`에 집중되어 있으며, 아이템의 `sellIn`과 `quality` 값을 매일 갱신하는 도메인 규칙을 다룹니다.

현재 구조상 `Item` 클래스는 외부 계약에 가까운 단순 데이터 클래스이며, 리팩토링 시 수정하지 않는 것을 원칙으로 삼는 것이 적절합니다.

## 2. 현재 기술 스택 확인

- 빌드 도구: Maven
- 테스트 프레임워크: JUnit 5
- 보조 테스트 라이브러리: ApprovalTests, Hamcrest
- 현재 `pom.xml`의 컴파일 설정: Java 8 (`maven.compiler.source`가 `1.8`)

주의 사항:
- 프로젝트 규칙상 Java 21을 목표로 한다면 `pom.xml`의 컴파일 버전 설정을 Java 21로 맞추는 작업이 필요합니다.
- JaCoCo는 요구 규칙에는 포함되어 있으나, 현재 `pom.xml`에서는 플러그인 설정이 확인되지 않습니다.

## 3. 핵심 도메인 규칙

리팩토링과 테스트 작성 시 다음 규칙을 항상 우선해야 합니다.

- `Item` 클래스는 수정하지 않습니다.
- 일반 아이템의 `quality`는 시간이 지날수록 감소합니다.
- `Aged Brie`는 시간이 지날수록 `quality`가 증가합니다.
- `Backstage passes`는 공연일이 가까워질수록 `quality` 증가량이 커지고, 공연일 이후에는 `quality`가 0이 됩니다.
- `Sulfuras`는 `sellIn`과 `quality`가 변하지 않는 예외 아이템입니다.
- `quality`는 일반적으로 0 이상 50 이하를 유지해야 합니다.
- `Sulfuras`는 `quality` 0~50 제한의 예외입니다.
- `sellIn`은 음수가 될 수 있으므로 0으로 강제 보정하면 안 됩니다.

## 4. 코드 관찰 사항

현재 `GildedRose.updateQuality()`는 모든 아이템 규칙이 하나의 반복문과 중첩 조건문 안에 섞여 있습니다.

주요 개선 지점:

- 아이템 이름 문자열이 여러 곳에 반복됩니다.
- `0`, `1`, `5`, `6`, `10`, `11`, `50` 같은 매직 넘버가 직접 사용됩니다.
- 조건문 중첩이 깊어 규칙별 동작을 한눈에 파악하기 어렵습니다.
- `Conjured` 아이템에 대한 로직은 현재 일반 아이템처럼 처리되는 것으로 보입니다.

권장 리팩토링 방향:

- 테스트를 먼저 보강한 뒤 리팩토링합니다.
- 아이템 이름과 품질 경계값을 상수로 분리합니다.
- 규칙별 메서드를 추출해 `updateQuality()`의 흐름을 단순화합니다.
- 동작 보존을 최우선으로 하고, 테스트 Green 상태에서만 작은 단위로 변경합니다.

## 5. 테스트 현황 및 보강 필요 사항

현재 테스트는 기본적인 샘플 테스트 중심이며, 도메인 규칙을 충분히 보호하기에는 부족합니다.

보강이 필요한 테스트:

- 일반 아이템의 `sellIn`, `quality` 감소
- `sellIn`이 0에서 -1로 넘어갈 때의 품질 변화
- `quality`가 0 아래로 내려가지 않는지 확인
- `quality`가 50을 초과하지 않는지 확인
- `Aged Brie`의 품질 증가와 상한값
- `Backstage passes`의 10일 이하, 5일 이하, 공연일 이후 동작
- `Sulfuras`의 `sellIn`, `quality` 불변
- `Conjured` 아이템 요구사항이 확정된 경우 일반 아이템보다 빠른 품질 감소

테스트 작성 규칙:

- Given-When-Then 구조를 사용합니다.
- 경계값 테스트에는 `quality` 0/50, `sellIn` 0/-1을 포함합니다.
- 여러 입력 조합은 `@ParameterizedTest`로 정리합니다.

## 6. 다음 작업 제안

1. `pom.xml`을 목표 기술 스택에 맞게 Java 21과 JaCoCo 설정으로 정리합니다.
2. 현재 동작을 보호하는 Characterization Test를 먼저 추가합니다.
3. 경계값 중심의 JUnit 5 테스트를 보강합니다.
4. 전체 테스트 Green 상태를 확인합니다.
5. 매직 넘버와 아이템명 상수화를 시작으로 작은 리팩토링을 진행합니다.
6. 규칙별 메서드 추출 후 테스트를 반복 실행합니다.

## 7. 결론

현재 프로젝트는 Gilded Rose 레거시 리팩토링을 진행하기에 적합한 상태이지만, 테스트 보호막이 충분하지 않습니다. 우선 도메인 규칙과 경계값 테스트를 보강한 뒤, `Item` 클래스 수정 금지와 기존 동작 보존 원칙을 지키면서 단계적으로 리팩토링하는 것이 가장 안전합니다.
