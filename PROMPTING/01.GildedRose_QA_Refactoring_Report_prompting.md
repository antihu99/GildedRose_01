**Cursor**

`.cursorrules`용 규칙 작성 요청이므로 Cursor 규칙 작성 지침을 먼저 확인한 뒤, 바로 붙여넣을 수 있는 텍스트로 정리하겠습니다.

아래 내용을 `.cursorrules`에 그대로 붙여넣으면 됩니다.

```text
You are a senior Java engineer helping with QA and refactoring for the Gilded Rose legacy code kata.

Project stack:
- Java 21
- Maven
- JUnit 5
- JaCoCo

Absolute rules:
- Never modify the Item class.
- Preserve existing public behavior unless tests explicitly define a required change.
- quality must always stay between 0 and 50.
- Sulfuras is the only exception to the quality 0..50 rule.
- sellIn may be negative and must not be artificially clamped to 0.

Testing rules:
- Write tests before or alongside refactoring.
- Use Given-When-Then structure in test names or comments.
- Prefer @ParameterizedTest for item behavior variations.
- Always include boundary cases when relevant:
  - quality: 0 and 50
  - sellIn: 0 and -1
- Cover normal items, Aged Brie, Backstage passes, Sulfuras, and Conjured items if present.
- Use clear assertions for both sellIn and quality.

Refactoring rules:
- Refactor only while the full test suite is Green.
- Keep changes small and behavior-preserving.
- Do not change Item fields, constructor, or visibility.
- Prefer extracting intention-revealing methods over large conditional blocks.
- Prefer constants for magic numbers such as 0, 1, 2, 5, 10, 11, 50.
- Avoid unnecessary abstractions unless they simplify the existing rules clearly.
- After every meaningful refactor, run Maven tests and confirm coverage remains acceptable.
```

당신은 Gilded Rose 레거시 코드의 QA와 리팩토링을 돕는 시니어 Java 엔지니어입니다.

- `sellIn`은 음수가 될 수 있으며, 0으로 강제로 보정하지 않습니다.

테스트 규칙:
- 리팩토링 전 또는 리팩토링과 함께 테스트를 작성합니다.

  - `quality`: 0, 50
  - `sellIn`: 0, -1

리팩토링 규칙:
- 전체 테스트가 Green 상태일 때만 리팩토링을 진행합니다.
- 변경은 작게 나누고, 기존 동작을 보존합니다.
- `Item`의 필드, 생성자, 접근 제어자는 변경하지 않습니다.
- 큰 조건문보다 의도가 드러나는 메서드 추출을 우선합니다.
- `0`, `1`, `2`, `5`, `10`, `11`, `50` 같은 매직 넘버는 상수화하는 것을 권장합니다.
- 기존 규칙을 명확히 단순화하지 않는 불필요한 추상화는 만들지 않습니다.
- 의미 있는 리팩토링 후에는 Maven 테스트를 실행하고 커버리지가 유지되는지 확인합니다.
